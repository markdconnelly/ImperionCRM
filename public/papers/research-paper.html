Data-structure decisions for an agentic operating system

The long-form rationale behind Imperion OS. Not what was built — the executive summary covers that — but why each data-structure decision is the one that holds up under real client data, real autonomy, and real audit. Every external influence is cited; nothing is invented.

Imperion · internal architecture paper · companion to docs/architecture/data-design-for-agents.md · external sources retrieved 2026-06-21

Abstract

The default way to add AI to a business application is to bolt an LLM + RAG onto a schema shaped for a human clicking a UI. It demos well and fails in production: the agent cannot tell where a fact came from or whether it is stale, which of three conflicting sources wins, where one user's memory ends and another's begins, or what it is allowed to do with what it retrieves. Imperion OS inverts the ordering — the data design came first, shaped for agents, and the UI renders over it. This paper defends six structural decisions that follow from that inversion: the medallion tiering, a single pinned vector space, concept-per-file semantics, a two-substrate personal memory with a temporal knowledge graph, two-axis row-level security, and a hybrid recall ranker.

1 · The ordering decision: data-first, not LLM-bolt-on

An operating system schedules processes over shared resources under governed access. Treat AI agents as the processes and "everything the business knows and can do" as the resource, and the data platform becomes the kernel: it must own provenance, meaning, retrieval, and permissions before any agent reasons or acts. That single reframe dictates every decision below. The contrast case — a vector index stapled to a human-form CRM with a chatbot reading off it — fails not because the model is weak but because the substrate cannot answer the questions an acting agent must ask first.

The structural claim of this paper: the failure modes of bolt-on agentic AI are data-structure failures, and each is closed by a specific, deliberate structural choice — not by a better prompt. The same instinct — that deterministic structure beats a fragile multi-agent framework — drives the action layer, where filesystem structure itself is the agentic architecture[8]; this paper concerns the data half of that thesis.

2 · Why medallion bronze / silver / gold

The medallion pattern progressively improves data quality across three layers, never losing the raw truth[1]. Imperion adopts it because the alternative — overwrite-in-place — discards provenance, and an agent acting on a fact must be able to answer "says who, and as of when?"

Bronze: lossless, per-source, provenanced

Every external fact lands as one physical table per (source × entity), storing the original payload verbatim; a union view stacks them with a source discriminator. Provenance is therefore structural — the table name is the source, so a row can never lose its origin — and bronze is an immutable reprocessing safety net: change a merge rule and silver rebuilds from bronze without re-fetching the external systems. Adding a source is additive (a new bronze table + one precedence entry), which is why the system scales to dozens of feeds without a downstream rewrite.

Silver: merged by explicit authority

Silver is one row per real-world entity, recomputed field-by-field, each field from the highest-precedence non-empty source (e.g. contact: website > autotask > itglue > m365 > apollo). The precedence order is the authority rule, documented per entity. This is where correctness lives or dies — matching keys, out-of-order arrivals, empty-vs-null — so it is tested and versioned, never hand-waved.

Gold: distilled, AI-ready

Gold is one citation-backed knowledge object per entity, derived from a known silver row, so a recalled summary traces back through silver to its bronze origin — RAG over a curated, provenanced substrate rather than a scraped pile, the single biggest lever on whether an agent hallucinates from noise. Drafts carry no embedding, so memory never recalls something unapproved.

Honest status. The gold schema and ~900 knowledge objects exist; embedding generation has not yet run (the vector column is empty), so semantic search is the contract and pipeline, not a live index at the time of writing.

3 · Why pgvector + Voyage voyage-3-large @ 1024d, one vector space

Three sub-decisions here, each defensible on its own.

Why pgvector (one store, not a bolt-on vector DB)

The embeddings live in the same PostgreSQL that holds the relational system of record. This is the "one store, many agents" shape that the Open Brain work argues for[7]: a single kernel every agent plugs into, instead of a relational DB plus a separate vector service that can drift, double the ops surface, and lose referential integrity between a vector and the row it summarizes. A gold embedding can join to its silver entity in one query because they share a database.

Why voyage-3-large

Voyage's voyage-3-large is a state-of-the-art general-purpose embedding model with Matryoshka-style dimension flexibility[3], paired with Anthropic-style contextual retrieval to cut retrieval failures by contextualizing chunks before embedding[4]. The model pair (Claude for generation, Voyage for embedding) is a settled contract, not a per-feature choice.

Why a single pinned 1024-dimension space

Every vector in the system is the same model at the same 1024 dimensions, polymorphic over any silver entity. The reason is brutally simple: vectors are only comparable within one space. Mix models or dimensions and cosine similarity between a contact and a ticket becomes meaningless. Pinning the dimension makes cross-entity recall valid by construction and makes re-embedding a deliberate, versioned migration rather than an accident.

4 · Why concept-per-file meaning (OKF)

Trustworthy data is necessary but not sufficient: an agent must also know what an entity means and which source wins. A bolt-on RAG system embeds rows and hopes the model infers meaning from text — which is exactly where confident, wrong answers come from. Imperion instead adopts Google's Open Knowledge Format — a vendor-neutral markdown + YAML-frontmatter spec for sharing curated knowledge with agents[2] — as a semantic layer of one concept file per silver entity.

Each file carries the entity's definition, its authority rule (the precedence order), its documented join paths, and a PII note. Three structural properties follow:

  • Retrieval you can trust, not vibes. Agents read the join path and which source is authoritative from a contract instead of guessing — the failure mode that produces plausible-but-wrong answers is removed at the source.
  • A PII firewall. The meaning layer is PII-free by design, so it can be shared, version-controlled, and even vectorized safely; specific personal values resolve against the live database, never a static file.
  • Freshness as correctness. A stale semantic layer lies with confidence, so staleness is made a CI failure — a PR that changes a silver table with a concept file must update that file in the same PR. The map cannot silently drift from the schema.

The same file a person reads is what the agent consumes — human-readable equals machine-readable, with no translation layer to drift. This is the meaning altitude of the same refinement DNA the medallion applies to data.

5 · Why a two-substrate personal memory with a temporal knowledge graph

A second brain needs to answer two different questions well: "what is true about me right now?" and "what did I actually say / decide, verbatim?" One store optimized for both does neither, so the personal tier uses two substrates.

The Synthesis Store — a temporal knowledge graph

Verbatim captures land in bronze; from them, a per-owner knowledge graph of entity–relation–object facts is synthesized. Each fact carries a validity window (valid_from / valid_to): a changed mind is a new fact that closes the prior one, never a destructive edit. This is the append-only-ledger discipline applied to personal knowledge — you can reconstruct what a brain believed at any past moment, which is both an audit property and the only honest way to model beliefs that change over time. Provenance is polymorphic: every fact cites the verbatim bronze turn (human capture or agent message) it was distilled from.

The Curated Vault — owner-edited markdown

Alongside the graph, each owner has a private Azure Blob "Curated Vault" of markdown they can edit directly, synced over HTTPS (no server, SMB, or VPN). This is the human-authored half of the second brain — durable, portable, plain-text — kept in sync with the synthesized graph by a background curator.

Verbatim recall, the MemPalace way

Recall is two-level, the pattern MemPalace demonstrates of storing history verbatim behind a semantic index[5]: hybrid-search the dense gold summaries first, then drill via a reference to the faithful verbatim turns only when the exact words matter. Search stays cheap and few; recall stays faithful and complete. Crucially this is expressed natively in the medallion (bronze raw + gold index + a reference) rather than as a bolt-on store with its own vector space — the consolidation-as-background-process idea that systems like claude-mem operationalize for session continuity[6], realized as data engineering.

6 · Why two-axis row-level security

An agent's memory must have boundaries enforced where the data lives, not where the app asks nicely. A bolt-on system enforces memory boundaries in application code — one missed check and a personal note surfaces in a shared answer. Imperion pushes the boundary into PostgreSQL row-level security along two axes:

AxisScopesEnforces
Companyrole → canon · company tierswhat a role may read across the shared estate
Personalowner → that owner's brain onlyone user's personal memory is invisible to another

A personal brain cannot leak into the company tier — it is a database invariant, not a convention. Cross-tier curation (personal → company) is done only by a privileged, non-BYPASSRLS service identity under an append-only god-view ledger, so even the curator's reach is itself scoped and audited. This is the OS ring/permission model applied to memory: the leak is made structurally impossible rather than conventionally discouraged.

7 · How the gold hybrid ranker scores

Vector-only retrieval is brittle: it misses exact identifiers (a ticket number, a domain) that lexical search nails, and it has no notion of recency or scope. The gold tier therefore ranks on four signals composed together, not similarity alone:

  1. Semantic similarity — cosine distance in the single 1024-d Voyage space; "about the same thing."
  2. Full-text (lexical) — PostgreSQL FTS for exact tokens an embedding blurs (IDs, names, error strings).
  3. Metadata — entity type, tier, source, draft-vs-published; a draft has no embedding and is invisible to recall.
  4. Temporal — capture/validity recency, so a current fact outranks a superseded one (the validity windows of §5 feed directly into rank).

Composing these is what makes recall both precise (lexical + metadata filtering) and relevant (semantic + temporal weighting). The contextual-retrieval guidance that motivates contextualizing chunks before embedding[4] applies here too: the substrate being ranked is curated and provenanced, so a high score means a verifiable result, not just a textually-similar one.

8 · Honest limits

A credible architecture names its edges:

  • Not one autonomous AI that "just runs the business." By design it is staged, contracted, and gated; "fully autonomous" is reframed as "fully automated with supervision you can dial."
  • Not real-time streaming analytics. The kernel is near-real-time (webhooks + on-demand refresh), not a stream processor.
  • The semantic layer is load-bearing. A stale OKF bundle would lie with confidence; the freshness gates are not optional polish.
  • The PII boundary is load-bearing — one mistake is a breach. Bundles and skills are PII-free by design; curation agents must aggregate/redact when walking production.
  • Recall is contract-complete, not yet hydrated. At the time of writing the personal substrate is provisioned but empty and embeddings have not been generated; the design is finished, the index is not.

Surfacing these is part of why the design is trustworthy — the same discipline that made provenance structural makes the limits explicit.
