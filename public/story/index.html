01
The spark

It started with a game

A friend was using Claude Code to build a prize game for RuneScape. The thought landed instantly: if it can do that, I can build a business application that delivers real value.

That night the target was a CRM to fix our sales pipeline — a single screen to stop losing deals in the cracks between tools. But the moment the first real data landed in a real database, the scope expanded on its own: if one agent could unify the pipeline, it could unify everything an MSP knows about a customer.

The ambition didn't stay small for long. Within days it was four repositories, one database, and a live product on Azure.

02
The problem

An MSP's knowledge is scattered and decaying

Running a managed-services business means knowing your customers cold. In practice that knowledge lives in ten disconnected places, and no one can see all of it at once.

Ten silos per contact — email, Teams, a text thread, a call recording, a LinkedIn comment, an Autotask ticket, someone's memory.
Reps walk in cold — hours of manual research and data entry an agent could have done overnight.
Profiling is legal risk — every text, call, and profile is a liability unless you can prove consent.
Tool sprawl — M365 + Kaseya + spreadsheets, re-keyed by hand, with no single source of truth.
03
What it is

One place to work. One agent to ask.

Imperion CRM is an AI-enabled operations platform that sits as an intelligence layer above Microsoft 365 and Kaseya — augmenting them, never replacing them. It manages the whole lifecycle of a managed-services customer, from the first ad they see to the quarterly business review years later: lead → qualification → proposal → onboarding → implementation → operational readiness → continuous success.

The promise is two-sided. One place to work — every account, contact, ticket, and conversation on one surface instead of ten. And one agent to ask — a single orchestrator that routes to specialist sub-agents behind the scenes, so the user never has to know which system an answer came from.

The defining go-to-market motion is assessment-led: a paid AI Security Readiness Assessment — six scored dimensions and a remediation roadmap — is the wedge that turns a cold prospect into a multi-year managed-services contract.

06
Security as a product feature

Built the way an MSP would secure it

Identity-first, least-privilege, fail-closed — the same posture Imperion sells, applied to its own platform.

Entra-only identityCertificate client auth; managed-identity DB access with no stored passwords.
Secrets in Key VaultPer-user OAuth tokens are referenced from Key Vault — never written to the database.
Per-client app, read-onlyEach client tenant consents a least-privileged read-only Graph app (ADR-0018); access still fails closed.
Consent ledgerAppend-only lawful-basis tracking; outbound is blocked unless consent is current (TCPA / CAN-SPAM / GDPR).
Golden-state + driftConditional Access, Intune, Defender, and Secure Score tracked against an approved baseline.
Human-gated grantsWidening any privilege or promoting a baseline is a deliberate, human-approved security event.

Trust, but verify

The governance is in the repo

Autonomy is only useful when it's accountable. Every repo carries the rules the agents must follow, and every decision is written down before it ships.

The velocity

A hallway complaint at noon. A shipped feature by night.

On 13 June 2026, Imperion spent the day at the Mahomet soda festival — a company outing, not a planning meeting. Somewhere between the root beer and the kettle corn, employees mentioned almost in passing that they were still logging their hours by hand. That was the entire brief. No ticket, no spec, no mockup — a complaint over a paper cup.

By the time the festival wrapped, time tracking was the app's top priority. A short Q&A turned the grumble into a real specification — who attests, who approves, how it reconciles against Autotask and QuickBooks — and the build began that same afternoon. The decision was written down first (ADR-0082, 6:06 PM). Then the feature was built end to end, in one continuous evening, while the git clock kept the receipts.

The bottleneck stopped being how fast people can type and became how fast a person can decide.
10
The method scales

One session became ten

The two-and-a-half-hour feature was one session driving one worktree. By that evening, that was the ceiling — not the model. The bottleneck had never been typing speed; it was how many independent changes one architect could keep coherent at once. So before bed, the method itself got an upgrade.

A short planning chat designed a concurrency contract: every session gets its own git worktree and its own dev port — s01→3001 … s10→3010, ten slots — shared hot files stay out of each other's lane, and the only counters GitHub can't hand out atomically, migration and ADR numbers, are claimed at merge instead of at authoring, so two parallel branches can never collide on the same number (ADR-0084). Ten sessions, one database, no stepped-on toes.

With the rails laid, ten coordinated sessions ran across all four repos at once. They didn't just polish time tracking — they carried it deeper (CFO payroll approval, the six-deviation reconciliation, a Time Efficiency BI section, inline admin correction) and stood up a whole second feature in the same night: employee expense tracking, end to end — ADR-0083, migrations 0088–0091, capture-to-reimbursement, with the matching backend writer, the bronze→silver merge, and the on-prem bulk pulls all filed alongside it.

Then the architect closed the laptop and went to bed — and the method had one more thing left to prove. Idea to a shipped feature in an evening, then one feature to a parallel two-feature build overnight, was the curve so far. The next question was sharper: could the whole thing be left entirely alone?

The bottleneck stopped being how fast people type and became how fast one person can decide. Parallelism only pays off if the parallel work can't corrupt itself — so the contract that prevents collisions shipped before the ten sessions did.

11
Planning is what makes AFK safe

He watched UFC 250, slept, and worked his day job. The robots didn't.

The parallel leap proved the method could fan out. The real test was whether it could run unattended — not one supervised evening at the keyboard, but a full day of the architect simply living his life. So that became the experiment. The fight card was on, then it was bedtime, then it was a full day at the actual MSP job. For most of a day, no one was watching the build.

It didn't slow down. In that ~24-hour window, governed sessions across all four repos opened, reviewed, and squash-merged roughly sixty pull requests — and none of it was filler. A near-complete project-management suite arrived in one stretch: sprints and backlog carry-over, baselines and planned-vs-actual, gantt / calendar / timeline views, task dependencies and subtasks, multiple assignees and watchers, @mentions, admin-definable custom fields, OKRs with rollup, workload and capacity planning, recurring tasks, form-to-task intake, kanban rich cards, and an in-app notification bell. Beside it, five more verticals broke ground — lead scoring, customer journeys, opportunity forecasting, e-signature, and conversational intelligence — plus DNS and security-posture drains, and the employee expense data-loop closed from capture to reimbursement.

None of it needed a human in the loop, because the loop is the repo. Every unit of work was an issue → a worktree → a branch → a reviewed micro-PR, judged against rules the agents re-read at the start of every session. And you can watch the plan defend itself in the git history: when two parallel branches reached for the same migration number, the concurrency contract caught the collision at merge and renumbered automatically — custom-fields 0102→0103, a 0104↔0105 cascade untangled, the freed gap back-filled — with no one awake to referee it. That is the entire thesis: the planning was paid up front, so the building could happen while the lights were off.

Unsupervised isn't ungoverned. The agents could be left completely alone precisely because the constraints weren't in anyone's head — they were committed to the repo: rule files, ADRs, a collision-proof numbering contract, and CI that fails closed. Plan once, then go watch the fight. Flipping anything to production data is still the only thing that waits for a human.

12
Four repos, one mind

The hard part isn't writing code. It's keeping four repos honest.

Speed in one repo is easy. The trap in a four-repo system is drift: the database moves in the front end, and three consumers — the backend's processes, the cloud pipeline's merge, the on-prem collectors — silently fall out of step. That's where most distributed builds quietly rot.

It didn't happen here, because the contract lives in the repo, not in anyone's head. Every repo carries a rule file the agent reads first; schema is proposed in one place (the front end owns migrations) and the siblings are declared consumers; one ADR number and one shared vocabulary thread through all four. The agent doesn't have to remember the architecture — it re-reads it every session and is held to it by CI.

The same time-tracking sprint shows it in motion: migrations 0085–0087 landed in the front end, and in the same push the matching epics and issues were filed across all three siblings — the backend writing idempotent Autotask Time Tickets on approval, the pipeline and on-prem collectors corroborating attendance. Four repos, never disagreeing about what a time_record means.

One agent held the whole system at once. What a team coordinates with meetings, the agent coordinates by reading the contract — so a schema change and its three downstream consequences ship as one coherent move, not a week of catch-up.

Betting on the frontier

Pioneering by default — three open standards, three layers

The point of this project was never just to ship fast; it was to find the edge of what's possible. So at each layer of the platform — data, AI, and now knowledge — Imperion adopted an emerging open standard early, before it was the safe consensus choice. Three layers, three bets, all in the open and auditable.
