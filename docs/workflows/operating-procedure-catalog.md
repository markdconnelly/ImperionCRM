# Imperion OS — Operating Procedure Catalog

The provable-coverage map of every **Operating Procedure** that runs the MSP — the
*outline layer* of the org recast (epic #1534). It enumerates the work, projects each
unit onto its one owning agent, and makes ownership holes, overlaps, and cross-agent
seams explicit. This is the **catalog (docs) layer**; the per-procedure `icm/`
workspaces are a later build phase.

- **Architecture decision:** [ADR-0133 — Operating Procedure catalog](../decision-records/ADR-0133-operating-procedure-catalog.md).
- **Canonical terms:** **Operating Procedure** and **Procedure Step** in [`CONTEXT.md`](../../CONTEXT.md) (Agent automation section).
- **Per-stream detail:** [`streams/`](streams/) — one file per value stream, full D6/D7 entry schema.
- **The org it projects onto:** the 26-agent roster (epic #1534, `imperion-os-org-recast`).

## How the catalog is built

**Value-stream-first spine (D2).** Procedures are generated by **value stream** — the
closed set of "what an MSP does" — then **projected onto the one owning agent**. Coverage
is provable against the stream; seams become explicit hand-off steps; ownership
holes/overlaps surface. The agent-first view (the epic tree) is a *derived projection* of
this spine, never the source.

**Sizing rule (D3).** One Operating Procedure has all four: (1) a named **trigger**;
(2) **exactly one owning agent** — a value stream that spans agents end-to-end is split at
each owner hand-off into per-agent procedures linked by explicit hand-off steps; (3) a
**terminal business outcome** (a nameable state change); (4) **runbook-sized** (~3–8
Procedure Steps; >~10 = two procedures; 1–2 = a step of a bigger one). Shared sub-steps
(resolve Client Mapping, send-for-esignature, post the work-note, margin-grounding) are
referenced Procedure Steps, not duplicated.

**Realization = the ICM Workspace is the SoR (D5).** A procedure with ≥1
`[automation]`/`[hybrid]` step is realized IN its ICM Workspace = live SoR; the human
`docs/runbooks/` runbook is a generated projection of it, never a competing home. A
fully-human procedure lives as a `docs/runbooks/<stream>/<proc>.md` document until it
graduates (gains an automatable step). The catalog index points at the canonical home.

**Entry schema (D6/D7).** Each procedure entry in the stream files carries: Name
(verb-noun) · Owner/Stream · Trigger · Terminal outcome · Procedure Steps (ordered,
tagged, hand-offs explicit) · Driving policy · Realization (workspace path | procedure-only)
· Autonomy ceiling (owner ladder ceiling + `always_gate` flags) · **Human-in-loop** (the
paired human + how involvement recedes with the dial, naming the dial-proof `always_gate`
steps) · Substrate deps (dormancy flags) · **subject** (`client | imperion` — Imperion is
a real managed client; dogfood is a parameter, not a duplicate procedure, D7).

## Catalog-wide design principles (D9 — apply to ALL ~174 procedures)

Every procedure's Human-in-loop field encodes these; they are not restated per entry.

- **P1 — Nova-native human co-working.** Every flow co-works with a human *through* Nova
  (the single orchestrator interface). No agent acts in a vacuum; Nova is the human's seat
  in every procedure.
- **P2 — Thought attribution up the chain.** Each sub-agent's reasoning is **ascribed back
  to the paired human** it is working with, surfaced up the hierarchy
  (worker → exec division → Nova → human). Rides the 3-level `agent_run`/`agent_message`
  ledger.
- **P3 — "Easy button" at every human gate.** Where a human gates the flow (approval /
  `always_gate` / a platform-gated act like a Meta ad boost + budget), the procedure
  **drives to the goal as far as possible** and hands the human a **one-click**
  resolution (generalizes the CONTEXT.md *Easy Mode* "clicking is deploying" to *every*
  gate). A gate is never "park and wait" — it is "prep everything + present the easy
  button." This is exactly why 01-B/01-C are real procedures (the agent does the full
  setup; the human clicks to actuate on the platform).
- **P4 — Notification routing.** An **urgent** issue pings a **dedicated chat**;
  otherwise the appropriate team member is **tagged in the shared Teams chat**. Teams is
  the human channel.

## Ownership rulings (D8)

The cross-cutting ownership calls applied across the streams:

- **Security Posture Management → Cyrus (SecOps).** Posture snapshot / secure-score /
  pillars / golden-state / DNS drift / client-standard measurement. **Vera owns the Client
  Security Standard; Celeste presents; human + Datto remediate.**
- **SLA management + SLA-aware assignment → Celeste (Customer Success).** Felix/Scout do
  the mechanical assignment; SLA adherence/ownership is Celeste's.
- **Connection-health monitoring → Vera (Platform / Governance)** — a platform function,
  OS-self-op (resolves the Ozzie-vs-Vera question).
- **Collections / AR-dunning → Finance (Audrey owns the procedure).** Audrey is
  read-only/L2, so the dunning **send** step is a human `[gui-step]`/`always_gate` in v1
  (Mark proxies); Audrey detects/drafts/ages, a human sends. (The named-agent ownership of
  the send remains a flagged Mark question — see Cross-cutting.)
- **AP intake → Finance (Audrey).** Books visibility; Vance owns vendor lifecycle.
- **Cross-client confidentiality → procedure DROPPED → the pool principle.** Clients are a
  **pool**: cross-correlate **internally** (a threat/signal at one client is checked across
  all — SecOps/Cyrus owns the pool cross-correlation procedure), but **never** allow
  client-facing bleed (RLS / data-class prevents inappropriate exposure). **Correlate,
  never bleed.**

## Routing model (D8)

Hierarchical **Nova → exec-agent → worker-agent** (confirmed). Each tier documents its
reasoning, presented up the chain (P2). The 5 division-orchestration procedures
(R2/D2/O2/S2/J2 in Stream 11) survive as load-bearing routing-within-division units; Nova
routes *across* divisions.
