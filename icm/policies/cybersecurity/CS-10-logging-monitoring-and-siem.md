## 1. Purpose

To define how security-relevant logs are generated, collected, retained, monitored, and analyzed so
threats are detected and responded to promptly across Imperion and client environments — and to
make every actor's behavior auditable. Because Imperion runs a hybrid workforce, this policy also
requires that **every governed agent action write an audit record**, so the autonomy framework is
provably enforced. It binds humans and agents identically.

## 2. Scope

**Who:** all Imperion employees, contractors, and AI agents whose actions touch Imperion or managed
systems. **What:** all Imperion and Imperion-managed systems, identities, networks, and cloud
workloads; the SIEM (Microsoft Sentinel) and the agent action ledger; and every Operating Procedure
that generates a security-relevant or agent-action event. This is a baseline policy inherited by
every procedure and every agent. Humans and agents are bound identically except where §5 sets
agent-specific evidence obligations.

## 3. Definitions

- **SIEM** — Security Information and Event Management; Imperion uses **Microsoft Sentinel**,
  integrated with **Microsoft Defender XDR**.
- **XDR** — Extended Detection and Response across endpoint, identity, email, and cloud apps.
- **SOAR** — Security Orchestration, Automation, and Response used to automate detection handling.
- **Telemetry** — security-relevant event data generated by systems and services.
- **Agent action ledger** — the 3-level `agent_run` / `agent_message` record that captures every
  governed agent action, its gauntlet result, and any human gate decision (top-umbrella §7).

## 4. Policy Statements

1. **Centralized logging architecture.** Security telemetry is centralized in Microsoft Sentinel,
   which ingests DNS, Microsoft Entra, and Azure activity logs, Defender XDR signals, and other
   relevant sources. Sentinel is integrated with Defender XDR for unified detection, investigation,
   and response. Where technically possible, security logs from mission-critical and client systems
   are forwarded to Sentinel.
2. **Required log sources and events.** At minimum the following are logged and retained:
   authentication events (success/failure), MFA challenges, and identity-risk events (Entra ID);
   privileged-account use and administrative actions, including delegated (GDAP) client-tenant
   actions; access to Restricted/Confidential data where feasible (CS-08); endpoint detections and
   response actions; email threats and DLP events; network/DNS activity and firewall events; cloud
   configuration changes; **AI system activity and agent actions** (CS-07); system start/stop and
   approved emergency access.
3. **Time synchronization and integrity.** System clocks are synchronized to an authoritative time
   source so events correlate accurately; logs are protected against tampering and unauthorized
   access; access to logging systems is least-privilege.
4. **Retention.** Security logs are retained for a minimum of **one year**, with at least 90 days
   immediately searchable, unless a longer period is required by contract, law, or regulation;
   client log retention follows the applicable client agreement.
5. **Monitoring, detection, and alerting.** Sentinel analytics rules and Defender XDR detections
   generate alerts triaged on a defined schedule; high-severity detections trigger the Technical
   Incident Response Program (CS-IR); detection content is tuned continuously; SOAR playbooks
   accelerate containment where appropriate.
6. **Continuous monitoring coverage.** Monitoring includes, but is not limited to: perimeter and
   firewall activity, intrusion-detection alerts, user and identity activity, identity-risk events,
   account-termination timeliness, application/database account activity, building-access anomalies,
   outbound DLP, inbound spam thresholds, AI system activity, and vulnerability-scan results.
   Exceptions or violations are recorded as security incidents with a documented risk assessment.
7. **Reporting.** Monitoring results and security posture are reported to the CISO at least annually
   and summarized for clients in quarterly business reviews (QBRs).

## 5. Application to Autonomous Agents

This policy is the agent evidence backbone. It is part of the baseline every agent inherits.

- **Every agent action writes an audit record.** For every governed action — a read of classified
  data, a proposal, a tool call, an actuation — the agent writes the 3-level `agent_run` /
  `agent_message` ledger entry: who/what acted, the `data_class` touched (without copying the
  sensitive value, CS-08), the gauntlet result, the dial level applied, and any human gate
  decision. Reasoning is attributed up the chain to the accountable human (top-umbrella P2). An
  action with no audit record is a violation, not a shortcut.
- **Forwarding to the SIEM.** Agent and AI-system activity is forwarded to Microsoft Sentinel
  (§4.2) so agent behavior is monitored alongside all other security telemetry.
- **Autonomy ceiling for monitoring agents.** Cyrus (SOC) operates detection/triage at L0–L2 and may
  reach **L4 — reversible-under-runbook** for containment actions that have an undo window;
  **containment is `always_gate`** at every dial level for destructive, identity,
  domain-controller, backup, or client-facing steps. Grace (GRC) is **L2 audit-and-recommend** over
  the logging program: she verifies coverage and retention, evidences gaps, and recommends — config
  or standard changes are `always_gate`.
- **`always_gate` actions (dial-proof floor).** A human decides, at every dial level, before an
  agent: disables or reconfigures a log source or the SIEM; alters retention; or executes a
  destructive/identity/domain-controller/backup/client-facing containment step. Each is presented
  as a one-click easy-button (top-umbrella P3).
- **Escalation & refusal.** An agent escalates a high-severity detection, a logging gap, a clock-
  skew or tamper signal, and any action it cannot audit. An agent **refuses** to take an action it
  cannot write a ledger record for, and **refuses** to disable logging regardless of dial.
- **Evidence.** The ledger itself is the evidence; the conformance sweep checks that every actuation
  has a matching audit record and that no agent action bypassed the gauntlet.

## 6. Roles & Responsibilities

| Actor | Responsibility |
| --- | --- |
| CISO (human owner) | Owns the program; receives annual monitoring/posture reporting; approves retention and SIEM changes. |
| Cybersecurity / SecOps (human) | Operates Sentinel/Defender, tunes detections, runs triage and IR. |
| Cyrus (SOC agent) | Detects, triages, and reports; L4 reversible-under-runbook containment; destructive/identity/backup/client-facing containment `always_gate`. |
| Grace (GRC agent) | Audits logging coverage, retention, and conformance; recommends; changes `always_gate` (L2). |
| All agents | Write the 3-level audit record for every action; forward activity to Sentinel; never act without an auditable record. |

## 7. Enforcement & Audit

Logging is enforced structurally (Sentinel forwarding, tamper-protected stores, least-privilege
access to logging systems, the mandatory ledger write) and verified continuously (analytics rules,
the conformance sweep matching actuations to audit records, eval goldens). The
[coverage-matrix](../coverage-matrix.md) proves every event-generating procedure is bound. A
violation — including an unaudited agent action or a disabled log source — parks the work and
escalates; repeated or high-severity violations lower the agent's dial or trip the kill-switch. For
humans, violations may result in disciplinary action up to and including termination.
